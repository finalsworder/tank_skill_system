# 坦克对战分层技能系统项目交接文档

## 1. 项目目标

本项目要构建一个**基于大模型高层指挥 + 强化学习低层执行**的坦克对战分层策略系统，用于在不超过 **3v3** 的坦克作战场景中：

1. 由 LLM 生成或选择**行动指令（技能）**及其训练配置。
2. 用 PPO 等强化学习方法训练这些行动指令对应的**低层执行策略**。
3. 在最终对战阶段，由 LLM 按固定周期为每辆坦克选择行动指令，并由对应的 PPO actor 执行。
4. 提高智能对手构建与测试的自动化程度。

这个系统的核心思想不是让 LLM 直接输出原子动作，而是：

- **LLM 负责高层任务分配 / 技能调用**
- **RL 负责低层动作控制**
- **环境负责仿真、观测组装、奖励计算、自动开火、记忆维护**

---

## 2. 项目当前采用的总体路线

### 2.1 分层结构

系统分为三层：

#### A. 底层：战场仿真层
负责：
- 地图与障碍物
- 坦克状态更新
- 雷达射线探测
- 炮线遮挡判定
- 自动开火
- 血量与摧毁
- 敌方最后出现信息记忆
- 出生点采样与合法性检查

#### B. 中层：技能训练层
负责：
- 用 JSON 配置单定义行动指令
- 定义任务参数、奖励函数、终止条件、动作维度、重置模板
- 根据某个行动指令自动组装 obs
- 用 PPO 训练对应策略
- 保存 checkpoint 与观测元信息

#### C. 上层：LLM 指挥层
负责：
- 读取可用行动指令集合
- 在完整对战阶段按**固定周期**做决策
- 为每辆坦克分配当前行动指令及参数
- 不直接输出原子动作，只调用技能

---

## 3. 最终目标场景与规模约束

### 3.1 最终应用场景

最终目标是 **3v3 坦克对抗**。

### 3.2 训练阶段规模约束

训练规模不超过 3v3，但不能只训练 1v1，否则泛化性不足。

因此训练时应支持：
- 1v1
- 2v2
- 3v3
- 混合模板随机采样

并且：
- 同一行动指令应对应**多个 reset template / 配置单**
- 每次 reset 时从多个模板中随机采样
- enemy_id / ally_id / 出生位置 / 朝向 / 初始发现状态 / 陪测策略都可变

## 4. 核心设计冻结点（目前已讨论较充分，建议后续继续沿用）

以下设计已经在讨论中逐步收敛，建议作为下一阶段实现的基线。

### 4.1 行动指令不是函数，而是 JSON 配置单

为了后续方便让 LLM 直接输出，行动指令配置必须采用 **JSON**，而不是 Python 函数形式。

一个行动指令 JSON 应至少包含：
- 指令名称
- 文字描述
- 允许的动作维度（`action_mask`）
- 使用的任务参数槽位
- 奖励函数及权重
- 终止条件
- reset 模板列表
- 观测组装配置
- 已训练模型路径（训练后补充）

### 4.2 固定任务参数槽位

为了避免系统过于通用导致不可控，任务参数固定为以下 4 个槽位：

- `target_x`
- `target_y`
- `ally_id`
- `enemy_id`

它们的含义：
- `target_x/target_y`：目标点机动类任务
- `ally_id`：掩护/跟随/协同类任务
- `enemy_id`：攻击/压制/跟踪类任务

参数值支持：
- 固定值
- 范围采样
- 离散集合采样

### 4.3 不把 raw id 直接喂给网络

虽然 JSON 里仍然可以配置 `ally_id` / `enemy_id`，但神经网络不直接吃原始编号。

当前推荐做法：
- 在 `ally_features` / `enemy_features` 中加入 `selected` 标志位
- 当前任务选择的 ally / enemy 在对应实体特征上标记 `selected=1`
- 网络只学“这个实体是任务选中对象”，而不是学编号映射

这样做比让网络直接学习 `enemy_id=2 -> 第三个实体` 更稳，更容易泛化。

### 4.4 task point 参数可直接并入 self_state

关于 `target_x/target_y`：
- 当前已收敛结论是：**可以直接并入 `self_state`**
- 如果某技能不使用这个参数，则训练时这两个值恒为 `0`
- 不再强制需要额外的 `task_param_mask`

原因：
- 对某个具体技能来说，如果参数不用，则它始终为 0
- 不同技能的策略不交叉训练时，这种“0 等价于未启用”是可接受的

### 4.5 动作空间简化，不学习开火时机

当前已决定：
- 低层策略不再学习显式 `fire`
- 环境在满足炮线条件时**自动开火**

因此当前低层动作只包括：
- 前进 / 停止 / 后退
- 车体左转 / 不转 / 右转
- 炮塔左转 / 不转 / 右转

即总共 3 个动作头。

### 4.6 `action_mask` 由 LLM 配置

行动指令可以限制允许学习的动作维度。

例如：
- 纯机动技能：只开速度和车体
- 纯瞄准技能：只开炮塔
- 攻击技能：开速度、车体、炮塔

这件事应由行动指令 JSON 中的 `action_mask` 指定，并允许后续由 LLM 决定。

### 4.7 固定周期重规划，不再事件触发

最终对战阶段，高层 LLM 的决策机制改为：

- **固定周期调用**，例如每 10/20/30 步重规划一次
- 而不是“某个 skill 终止时立即触发”

这样更稳定，也更便于批处理多辆坦克。

### 4.8 PPO actor 在对局阶段不动态创建

最终对局阶段不应该每次指令变化时动态创建 actor。

正确做法：
- 在对局开始时一次性加载所有已注册行动指令的 actor
- 放入 `ActorPool`
- 同一个行动指令的 actor 可以同时为多辆坦克服务
- 以 batch 形式输入 obs，统一前向

### 4.9 训练完成后必须保存“观测组装方法”

因为不同技能最终可能使用不同的 obs 组合方法，所以每个行动指令在训练后注册时必须保存：
- checkpoint
- 行动指令 JSON
- observation metadata
- 网络输入维度说明
- 动作头定义

这样最终对战时，env 才知道如何为某辆正在执行某技能的坦克自动组装 obs。

## 5. 环境规则设计（已讨论并形成的版本）

## 5.1 地图与障碍物

- 地图为二维平面，默认 `1000 x 1000`
- 障碍物可以固定布置
- 障碍物布局可以与最终对战场景一致
- reset 时单位生成必须避开障碍物，不能出生在障碍物内部

## 5.2 坦克碰撞/遮挡规则

当前讨论后收敛为：

### 运动碰撞
- 坦克与坦克之间可以不再有实体碰撞体积
- 障碍物仍然阻挡运动
- 出生时仍需避免单位之间初始重叠

### 炮线遮挡
- 障碍物会遮挡炮线
- **存活的敌方坦克**会遮挡炮线
- 队友不会遮挡炮线
- 被击毁的坦克不会遮挡炮线

## 5.3 雷达观测

原始环境中有雷达射线，后来要求恢复并改造为：

- 雷达输出不再是“命中类型”
- 改成“距离”
- 检测对象包括：
  - 障碍物
  - 地图边界

即：每根射线返回最近命中目标的距离。

当前推荐：
- 72 根射线
- 归一化到最大距离1000
- 放进 `self_state`

## 5.4 视野与记忆

环境中需要维护敌方单位的记忆：

- 敌人被发现后，如果重新藏起来，仍然保留最后一次出现信息
- 这会影响 obs 中 `enemy_features` 的：
  - `visible`
  - `memory_valid`

此外，需要支持在 reset 时指定：
- 某个敌方单位是否“已经被发现过”
- 这会直接影响以 `enemy_id` 为目标的技能是否有可训练信号

## 5.5 血量机制

已新增需求：
- 坦克初始 `hp = 5`
- 每命中一次 `hp -= 1`
- `hp == 0` 时被击毁
- 需要将hp也加入自己和ally enemy的特征向量中
这比“一枪即死”更适合训练技能，尤其是攻击/掩护类技能。

## 6. 观测设计（当前项目建议版本）

当前项目观测采用**实体中心 + 雷达 + 任务参数点坐标**的方案。

### 6.1 网络输入结构

当前建议输入：

```python
{
    "self_state": ...,
    "ally_features": ...,
    "ally_mask": ...,
    "enemy_features": ...,
    "enemy_mask": ...,
}
```

### 6.2 `self_state`

包含：
- 当前步数 / 最大步数
- 自身坐标
- 车体朝向
- 炮塔朝向
- 速度
- 开火冷却
- 当前是否被敌方持续观测
- `target_x`
- `target_y`
- 一圈雷达距离

说明：
- 如果当前技能不用 `target_x/target_y`，则这两个值固定为 0

### 6.3 `ally_features`

每个队友的特征建议包括：
- 坐标
- 朝向
- 炮塔朝向
- 存活状态
- 是否被观测
- `selected` 标志

### 6.4 `enemy_features`

每个敌人的特征建议包括：
- 坐标（当前可见则为真实位置，不可见但记忆有效则为最后出现位置）
- 朝向
- 炮塔朝向
- 存活状态
- `visible`
- `memory_valid`
- 与自身炮塔瞄准误差
- `selected` 标志

### 6.5 可变实体输入处理

因为队友和敌人数量是可变的，网络应使用：
- `ally_mask`
- `enemy_mask`
- 自注意力或 attention pooling 进行聚合

并且要特别注意数值稳定性：
- 如果某个 batch 样本里没有任何有效队友/敌人
- attention 模块必须返回 0 向量
- 不能把“全 mask”的序列直接送进 MultiheadAttention

否则会产生 `NaN`

## 7. 奖励函数体系（建议冻结为受控集合）

奖励不允许由 LLM 直接编码，而应由系统预定义一组奖励函数，LLM 只负责选择和赋权。

## 7.1 与目标点绑定的奖励

自动绑定 `target_x/target_y`：
- `progress_to_target_point`
- `arrive_target_point`
- `heading_align_to_target_point`

## 7.2 与选中敌方绑定的奖励

自动绑定 `enemy_id` / `selected enemy`：
- `hit_selected_enemy`
- `destroy_selected_enemy`
- `aim_align_selected_enemy`
- `selected_enemy_visible`
- `selected_enemy_memory_valid`

## 7.3 与选中友方绑定的奖励

自动绑定 `ally_id` / `selected ally`：
- `ally_selected_hit`
- `ally_selected_alive`
- `stay_near_selected_ally`

## 7.4 通用奖励

与具体目标无关：
- `hit_any_enemy`  （新增，自由射击命中任意敌方）
- `self_hit`
- `self_destroyed`
- `discover_any_enemy`
- `discover_selected_enemy`
- `under_observation`
- `continuous_under_observation`
- `collision_penalty`
- `time_penalty`

说明：
- `hit_any_enemy` 是为了避免攻击类技能过于僵硬，只对指定 id 有奖励
- 也有利于训练更具泛化性的攻击行为

---

## 8. 终止条件设计

每个行动指令 JSON 都应包含自己的终止条件。

建议分三类：

### 成功条件
例如：
- 到达目标点
- 命中指定敌方
- 击毁指定敌方
- 发现任意敌方
- 发现指定敌方

### 失败条件
例如：
- 自己被击毁
- 指定友方被击毁（掩护任务）

### 截断条件
例如：
- 达到最大步数

运行时建议：
- 训练阶段严格使用这些终止条件
- 对战阶段则主要靠固定周期重规划，不强依赖 skill terminal 立刻重规划

---

## 9. 行动指令 JSON 的角色

一个行动指令 JSON 是整个系统的核心配置单。

它描述：
1. 这是一个什么技能
2. 它用什么动作维度
3. 它依赖哪些任务参数
4. 它用哪些奖励函数
5. 它如何终止
6. 它在什么样的 reset 模板下训练
7. 它训练后对应哪个模型
8. 它需要什么样的观测组装方式

换句话说：

**行动指令 JSON = 训练规范 + 运行规范**

---

## 10. Reset 模板设计

一个技能不能只对应一个固定训练场景，而要有多个 JSON 配置模板。

每个 reset template 应包含：
- 单位规模（1v1 / 2v2 / 3v3）
- 我方/敌方/队友出生区域
- 朝向范围
- 固定障碍物场景
- 初始发现状态
- 陪测脚本策略
- 参数采样规则

每次训练 reset：
- 从该技能的多个模板中随机选一个
- 再在模板内部随机采样

这样可以提升泛化性，避免：
- 只会在固定 enemy_id 上工作
- 只会在固定坐标点上工作
- 只会在 1v1 上工作

---

## 11. 陪测脚本策略（训练环境中的其它单位策略）

训练某个技能时，其他单位不应该都静止不动，而应由预定义脚本控制。

当前已经讨论出的固定策略集合：

- `idle`：原地不动
- `aim_closest`：原地瞄准最近可打敌方，自动开火
- `attack_point`：向目标点机动，若能攻击则自动瞄准并开火

这些脚本策略由 reset template 一次性配置好，在整个 episode 中保持不变。

用途：
- 让训练环境更接近最终对战条件
- 为进攻/规避/掩护类技能提供背景交互

## 12. PPO 训练逻辑

## 12.1 训练默认视角

训练默认从**红方视角**进行。

但更本质地说：
- PPO 环境内部不区分红蓝语义
- 只区分“self / ally / enemy”

也就是说，训练代码逻辑应尽量是**敌我相对视角**。

## 12.2 并行采样

训练阶段应支持：
- 多 CPU 核
- 多环境并行采样
- headless 模式

这对于 PPO 样本吞吐很重要。

## 12.3 两个示例训练技能

当前明确要求至少提供两个训练脚本示例：

### move 指令
目标：
- 学会移动到目标点

特点：
- 重点使用 `target_x/target_y`
- 一般不依赖 `enemy_id`

### attack 指令
目标：
- 面向目标敌方实施接近、瞄准、攻击

特点：
- 需要 `enemy_id`
- 可能也用 `target_x/target_y` 做辅助机动
- 可利用 `hit_selected_enemy` 和 `hit_any_enemy`

## 12.4 训练脚本风格要求

主程序要求：
- 不用 `argparse`
- 用大写全局变量定义参数
- 不定义 `main()`
- 直接在 `if __name__ == '__main__':` 下写主逻辑
- 尽量避免多余抽象与冗余设计
- 避免用 `try` 去吞掉异常，方便暴露真实报错

## 13. 已发现并修复的重要问题：NaN

在使用 attention 聚合可变实体输入时，曾出现训练时：

```python
Categorical(logits=...) -> logits 全为 NaN
```

根因：
- 某些 batch 样本中没有有效队友或有效敌人
- `key_padding_mask` 全为 `True`
- 仍把该样本送进 `MultiheadAttention`
- attention softmax 输出 NaN

修复思路：
- 对 attention pooling 做保护
- 若某个 batch 行完全没有有效实体，则该行 pooled embedding 直接返回全 0
- 不送进 attention

这个修复是必须保留的，否则：
- 1v1 场景没队友时
- 或混合规模训练时
- 很容易再次出现 NaN

## 14. 最终对战阶段的运行逻辑

这是项目最终想实现的关键闭环。

## 14.1 启动阶段

对战开始前：
- 初始化最终 3v3 场景
- 固定障碍物布局
- 加载所有已训练行动指令
- 一次性创建所有 actor，放进 `ActorPool`
- 初始化 LLM agent（外部传入 api 对象）

## 14.2 高层决策

每隔固定周期（例如 20 步）：
- 对每辆由 LLM 控制的坦克构造战场摘要
- LLM 从“已注册行动指令集合”中为它选择一个技能
- 并输出该技能对应参数（例如 target_x/target_y/enemy_id/ally_id）

## 14.3 低层执行

对于每辆坦克：
- 根据当前技能定义自动组装 obs
- 将 obs 送入对应的 PPO actor
- 由 actor 输出机动/转向/转炮塔动作
- env 自动开火

## 14.4 多辆坦克并行

同一个行动指令的 actor 不动态创建，而是：
- 复用同一个 actor
- 支持 batch 输入多个坦克的 obs

例如：
- 0 号和 2 号坦克此时都执行 `attack`
- 则它们的 obs 可以拼成 batch
- 一次前向完成

这就是为什么最终需要 `ActorPool`。

---

## 15. LLM API 设计要求

旧工程中 `llm_agent` 里有硬编码 API key 或模型对象，这一设计不再适用。

现在已明确要求：
- **api 对象由外部创建**
- 再传入 `llm_agent`

正确关系应当是：

```python
api = SomeLLMAPI(...)
agent = LLMAgent(api=api, action_registry=registry)
```

而不是：
- `llm_agent` 内部自己写死 key
- `llm_agent` 内部自己创建具体 client

这样做的好处：
- 便于切换不同模型/API
- 配置和逻辑分离
- 更方便测试

## 17. 当前项目仍需重点补强的部分

虽然已经明确了设计和一部分代码方向，但如果在新对话中继续推进，建议优先关注以下事项。

### 17.1 统一并稳定代码工程
过去多轮生成中，曾出现：
- 工程目录不完整
- 某些模块缺失
- 某些链接失效

新一轮讨论中应优先要求：
- 生成一套稳定、完整、可下载的工程目录
- 每次改动后确保 zip 可用

### 17.2 让 JSON 配置单真正驱动训练与运行
不是“有 JSON 示例”，而是：
- 训练器真正读取 JSON
- env reset 真正由 JSON 模板驱动
- runtime 真正由 JSON 中的 obs 配置 / reward 配置驱动

### 17.3 把最终对战 runtime 闭合
目前最核心但仍需重点确认的是：
- `battle_runtime`
- `LLMAgent`
- `ActorPool`
- `ActionRegistry`

是否已经完整闭环。

### 17.4 统一旧工程与新工程的几何规则
旧工程中的 `utils.py` 有较完整的几何工具函数，可用于对齐：
- 射线与矩形求交
- 线段遮挡
- 角度偏差
- 矩形重叠

新阶段应考虑把这些逻辑吸收到统一 `geometry.py` 中，避免环境规则漂移。

## 18. 推荐的下一阶段推进顺序

在新对话中，建议按以下顺序继续推进：

### 第一步：冻结 JSON schema
先把：
- ActionSpec JSON
- ResetTemplate JSON
- SceneSpec JSON

三个 schema 最终定稿。

### 第二步：冻结 env 接口
明确：
- reset 输入什么
- step 输出什么
- obs builder 如何绑定 action spec

### 第三步：冻结训练链
确保：
- move / attack 两个技能都能用 headless 多核训练
- 训练日志稳定
- NaN 不再出现

### 第四步：冻结运行链
确保：
- ActorPool 工作正常
- LLM 固定周期调用技能
- 一辆或多辆坦克可并行执行不同技能

### 第五步：扩展技能与模板
在前面链条稳定后，再扩展更多技能，如：
- recon
- evade
- cover ally
- flank
